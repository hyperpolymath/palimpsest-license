== Claim 1: File-Level Copyleft Built on Mozilla Public License 2.0

[quote, README]
____
Built on Mozilla Public License 2.0.
____

*How it works:*
The canonical license text lives at `legal/PALIMPSEST-MPL-1.0.txt` (and a
versioned copy at `v1.0/LICENSE.txt`). The PMPL-1.0 PREAMBLE states that it
"incorporates the full text of Mozilla Public License 2.0 by reference" — the
base MPL-2.0 terms apply except where modified by two Exhibits. Exhibit A adds
ethical use guidelines (cultural sensitivity, emotional lineage preservation,
AI training obligations). Exhibit B adds optional post-quantum cryptographic
provenance signatures using NIST-approved algorithms (ML-DSA, SLH-DSA, FN-DSA).

File-level copyleft is inherited directly from MPL-2.0: modifications to a
PMPL-licensed *file* must be released under the same licence, but code in
separate files may remain proprietary — matching MPL's "weak copyleft" model.

The SPDX identifier `PMPL-1.0-or-later` is used throughout the hyperpolymath
account. All tooling (including CI workflows and the `pmpl-audit` CLI) checks
for this exact identifier string.

*Honest caveat:*
PMPL is not an OSI-approved license. Ecosystems that require OSI-approved
identifiers (Ada/GNAT, Chrome/Firefox extension stores) must fall back to
`MPL-2.0`. The account's global CLAUDE.md documents this fallback policy.

== Claim 2: Compliance Tooling — Sign, Verify, Audit

[quote, README]
____
Quick Start: curl -O https://raw.githubusercontent.com/hyperpolymath/palimpsest-license/main/v1.0/LICENSE.txt
— Add SPDX headers to your source files. That's it!
____

*How it works:*
Three Rust CLI tools sit under `tools/`:

- `pmpl-sign/` — Signs a file or directory with a post-quantum provenance
  signature (Exhibit B). Writes a `.pmpl-sig` sidecar file.
- `pmpl-verify/` — Verifies sidecar signatures and checks the provenance chain.
- `pmpl-audit/` — Walks a repository tree and reports compliance. Entry point:
  `tools/pmpl-audit/src/main.rs`. Uses `clap` for argument parsing, `walkdir`
  for directory traversal, and `regex` for SPDX header matching. Outputs a
  structured `AuditResult` in text or JSON format reporting:
  files checked, files with headers, files missing headers, and whether the
  run passed.

All three tools are `#![forbid(unsafe_code)]`. Build with `just build` in the
repo root (delegates to `cargo build --release` per each tool's `Cargo.toml`).
Run checks with `just validate`.

The `deno.json` at root drives template and documentation tooling written in
Deno/ReScript; it is separate from the Rust tools.

*Honest caveat:*
Provenance signature verification (`pmpl-verify`) requires the ML-DSA / SLH-DSA
cryptographic libraries, which have not yet been integrated. The verify tool
framework exists but signature checks are stubbed pending library selection.
See `PROOF-NEEDS.md` and `TEST-NEEDS.md`.
